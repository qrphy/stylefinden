Detail sayfaları neden var, Sanity nasıl çalışır, içerik ekleme'den sayfaya render'a kadar her adımın açıklaması.

1. Büyük Resim: Ne Nedir, Ne Yapar?

Projenin içerik mimarisi iki katmana ayrılıyor: yapısal navigasyon katmanı (sayfalardaki başlıklar, açıklamalar, filtreler, FAQ'lar) ve dinamik içerik katmanı (outfitler, aksesuarlar, saç stilleri, blog yazıları).

Katman Nerede yaşıyor? Kim günceller? Değişme sıklığı
Yapısal / Navigasyon
Sayfa başlıkları, kategoriler, SEO metinleri, FAQ		 Kod içinde (TypeScript) Developer Nadiren (sezonluk)
Dinamik İçerik
Outfit, aksesuar, saç stili, blog yazısı		 Sanity CMS (bulut) İçerik editörü, sahibi Sürekli (günlük/haftalık)

Sanity, bu dinamik içerik katmanını yönetmek için kullanılıyor. Kod değiştirmeden, deploy etmeden, bir arayüzden (Studio) içerik ekleyip değiştirebiliyorsun.

2. Detail Sayfaları Neden Var? ([slug] Mantığı)

Temel soru: Outfit listesinde 50 outfit varsa, her outfit için ayrı bir sayfa dosyası oluşturmak mümkün mü? Hayır. İşte bu yüzden dynamic route ve slug kavramları geliyor.

Slug Nedir?

Slug, bir içeriği URL'de temsil eden benzersiz kısa metindir. "Classic Bob" başlıklı bir saç stili için slug: classic-bob olur. Bu slug URL'de kullanılır: /hairstyles/classic-bob

Static Route vs Dynamic Route

Yaklaşım Örnek Sorun
Her sayfa için ayrı dosya outfits/classic-bob.tsx
outfits/boho-dress.tsx		 100 outfit = 100 dosya. Deploy gerektirir. Sürdürülemez.
Dynamic route [slug] outfits/[slug]/page.tsx Yok. Tek dosya, sonsuz sayıda içeriği render eder.

src/app/outfits/[slug]/page.tsx dosyası, URL'deki [slug] kısmını alır, Sanity'ye sorar: "Bu slug'a sahip bir outfit var mı?" Varsa sayfayı oluşturur, yoksa 404 döner. 

// src/app/outfits/[slug]/page.tsx
export default async function OutfitPage({ params }) {
  const { slug } = await params

  // Sanity'ye sor: bu slug'a sahip outfit var mı?
  const outfit = await client.fetch(OUTFIT_QUERY, { slug })

  // Yoksa 404
  if (!outfit) notFound()

  // Varsa sayfayı render et
  return <main>...</main>
}

generateStaticParams — Build Zamanında Sayfa Üretimi

Next.js, hangi slug'ların var olduğunu önceden bilmek ister — bu sayfaları build sırasında statik HTML olarak üretir (Static Site Generation). Bunun için generateStaticParams fonksiyonu kullanılır: 

export async function generateStaticParams() {
  // Sanity'den tüm outfit slug'larını çek
  const slugs = await client.fetch(
    `*[_type == "outfit"]{"slug": slug.current}`
  )
  // Next.js'e hangi sayfaları önceden üretmesi gerektiğini söyle
  return slugs  // [{ slug: "casual-chic" }, { slug: "boho-dreams" }, ...]
}
Performans Açısından Önemi

Build zamanında üretilen sayfalar, kullanıcı isteğinde Sanity'ye gitmeye gerek kalmadan hazır HTML olarak sunulur. Bu 0ms sunucu süresi demektir — sayfa CDN'den anında gelir. Yeni bir outfit eklendiğinde Next.js, sadece o yeni sayfa için yeniden üretim (ISR) yapar.

Adım 1 — Sanity Studio (İçerik Yönetim Arayüzü)

Proje içindeki /studio adresi, Sanity'nin tam özellikli CMS arayüzünü açar. Burada:

  • Yeni outfit eklenebilir, mevcut outfit düzenlenebilir
  • Görsel yüklenebilir (Sanity kendi CDN'ine kaydeder)
  • İlişkiler kurulabilir: "Bu blog yazısıyla ilgili outfitler" gibi
  • Draft (taslak) ve Published (yayınlandı) ayrımı yapılabilir
Schema Nedir?

Schema, bir içerik türünün hangi alanları taşıyacağını tanımlar. src/sanity/schemaTypes/outfit.ts dosyası, Studio'da outfit oluştururken hangi alanların görüneceğini belirler: title, image, style, season, occasion, pieces, tags, featured… 

// src/sanity/schemaTypes/outfit.ts — Schema tanımı
export const outfit = defineType({
  name: 'outfit',
  type: 'document',
  fields: [
    defineField({ name: 'title',    type: 'string'                    }),
    defineField({ name: 'image',    type: 'image', options: { hotspot: true } }),
    defineField({ name: 'style',    type: 'string', options: { list: [...] }   }),
    defineField({ name: 'season',   type: 'string', options: { list: [...] }   }),
    defineField({ name: 'featured', type: 'boolean'                   }),
  ]
})

Adım 2 — Content Lake (Sanity'nin Veri Tabanı)

Studio'da "Publish" butonuna basıldığında içerik, Sanity'nin bulut altyapısına (Content Lake) kaydedilir. Bu, Sanity'nin barındırdığı bir API + veritabanı kombinasyonudur.

  • Proje ID: 29dp442n
  • Dataset: production
  • Tüm içerik bu bulut deposunda JSON formatında yaşar
  • Görüntüler Sanity CDN'inde barındırılır: cdn.sanity.io

Adım 3 — GROQ Sorgusu (Veriyi Çekme Dili)

GROQ (Graph-Relational Object Queries), Sanity'ye özgü sorgulama dilidir. SQL'e benzer ama JSON belgeleri için optimize edilmiştir. 

// Tüm outfit'leri en son eklenenden başlayarak çek
*[_type == "outfit" && defined(slug.current)] | order(_createdAt desc) {
  _id, title, "slug": slug.current, image, style, season, occasion, tags
}

// Sadece "autumn" sezonundaki outfitler
*[_type == "outfit" && season == "autumn"] {
  _id, title, "slug": slug.current, image, style
}

// Belirli bir slug'a sahip tek outfit
*[_type == "outfit" && slug.current == $slug][0] {
  _id, title, description, image, style, season, pieces, tags
}

Tüm GROQ sorguları src/lib/queries.ts dosyasında merkezi olarak tanımlanmıştır. defineQuery() fonksiyonu, TypeScript tip çıkarımı sağlar — yani sorgu sonuçları otomatik olarak doğru TypeScript tipini alır.

Adım 4 — Next.js Server Component Fetch

Next.js'in server component'leri, sayfa sunucuda oluşturulurken direkt olarak Sanity'ye bağlanır ve veriyi çeker. Kullanıcıya ulaşan sayfa zaten hazır HTML'dir — tarayıcı hiç API çağrısı yapmaz. 

// Sayfa server'da çalışır, kullanıcı görmez
export default async function FeaturedOutfits() {
  const outfits = await client.fetch(
    HOME_OUTFITS_QUERY,
    {},
    { next: { revalidate: 3600, tags: ['outfit'] } }
  )

  // Sanity'den gelen veriyle render et
  return (
    <section>
      {outfits.map((outfit) => (
        <a href={`/outfits/${outfit.slug}`}>
          <ImgPlaceholder src={urlFor(outfit.image).width(600).url()} />
          <h3>{outfit.title}</h3>
        </a>
      ))}
    </section>
  )
}

Adım 5 — urlFor() ile Görsel Optimizasyonu

Sanity görselleri ham formatta depolar. urlFor() fonksiyonu, ihtiyaca göre boyut, format ve kalite parametreleriyle optimize URL üretir: 

// 600x800, WebP formatında, %80 kalitede
urlFor(outfit.image).width(600).height(800).format('webp').url()
// → "https://cdn.sanity.io/images/29dp442n/production/abc123-600x800.webp"

// 400x400 kare kırpma
urlFor(item.image).width(400).height(400).fit('crop').url()

Bu URL'ler next/image'in src prop'una verilir. Next.js bu görüntüleri ek olarak kendi CDN'inden de sunar.

4. Caching Stratejisi ve Performans

Sanity her sayfa yüklemesinde sorgulanmaz. Next.js, veriyi birden fazla katmanda önbelleğe alır:

Katman Ne yapar? Ne zaman geçersiz olur?
Build Cache Build sırasında tüm bilinen sayfalar statik HTML olarak üretilir Yeni deploy
ISR (Incremental Static Regeneration) revalidate: 3600 — sayfa en fazla 1 saatte bir yenilenir Her saat başı veya tag ile tetikleme
Tag-Based Revalidation tags: ['outfit'] — Sanity webhook'u tetiklediğinde sadece ilgili sayfalar yenilenir Sanity'de içerik publish edildiğinde
CDN Cache Vercel Edge Network, statik varlıkları dünya genelinde dağıtır Revalidation sonrası
Kullanıcı Deneyimi Açısından

Kullanıcı /outfits/casual-chic sayfasına girdiğinde, Sanity'ye hiçbir istek gönderilmez. Sayfa zaten build'de üretilmiş ve CDN'de hazır bekliyor. Yanıt süresi: ~50ms. Sanity olmayan bir durumda server-side render yapılsaydı bu 200–500ms olurdu.

5. İçerik Ekleme — Adım Adım Akış

Bir outfit eklemek için ne gerekiyor, hiç kod yazmadan nasıl yapılıyor:

01

Studio'ya git

https://stylefinden.com/studio adresine git. Sanity hesabınla giriş yap.

02

Yeni Outfit oluştur

Sol menüden "Outfit" → "New Outfit" tıkla. Başlık, görsel, stil, sezon, occasion alanlarını doldur.

03

Slug otomatik oluşur

Başlığı yazdığında slug otomatik üretilir: "Elegant Black Dress" → elegant-black-dress

04

Publish et

"Publish" butonuna bas. İçerik anında Content Lake'e kaydedilir.

05

Sayfa otomatik oluşur

Next.js ISR devreye girer. En geç 1 saat içinde (veya webhook ayarlandıysa anında) /outfits/elegant-black-dress sayfası hazır olur. Aynı zamanda ana sayfadaki Featured Outfits bölümünde de görünür.

06

Güncelleme de aynı şekilde

Studio'da herhangi bir alanı değiştir, tekrar Publish et. Sayfa bir sonraki revalidation'da yeni içerikle güncellenir.

8. Sanity'ye Özgü Kavramlar — Sözlük

Kavram Açıklama
Content Lake Sanity'nin bulut veritabanı. Tüm içerik JSON olarak burada saklanır.
Studio Sanity'nin açık kaynak CMS arayüzü. /studio adresinde çalışır.
Schema İçerik türünün alan yapısını tanımlayan TypeScript kodu.
GROQ Sanity'nin sorgu dili (Graph-Relational Object Queries). Sanity'ye özgü, SQL benzeri.
Document Sanity'de bir içerik birimi. Her outfit, her blog yazısı bir "document"tır.
Draft / Published Her document'ın iki hali olabilir: taslak (draft) ve yayınlandı (published).
Reference İki document'ı birbirine bağlayan alan. Blog yazısı → ilgili outfitler gibi.
Portable Text Sanity'nin rich-text formatı. Blog yazısı gövdesi bu formatla saklanır.
Hotspot Görsel kırpma için odak noktası tanımlama. Farklı aspect ratio'larda görsel güzel görünür.
defineQuery next-sanity'nin GROQ sorgusuna TypeScript tipi ekleyen sarmalayıcısı.
ISR Incremental Static Regeneration. Sayfaları build'de üretip periyodik yenileme.
Tag Revalidation Next.js cache etiketlerine göre seçici sayfa yenileme. Sadece değişen içerik yenilenir.

9. Tam Akış — Görsel Özet

İçerik Ekleme Akışı:
Studio (tarayıcı)  →  Sanity Content Lake (bulut)  →  Webhook tetiklenir (opsiyonel)  →  Next.js ISR  →  Yeni statik sayfa üretilir  →  Vercel CDN dağıtır  →  Kullanıcı görür

Sayfa Yükleme Akışı:
Kullanıcı /outfits/casual-chic girer  →  Vercel CDN (önbellekte var mı?)  →  Evet: Hazır HTML anında döner (~50ms)  →  Hayır: Next.js server oluşturur & Sanity'den çeker (~300ms)  →  Sonraki istek için önbelleğe alınır

Görsel Akışı:
Sanity'de görsel upload edilir  →  cdn.sanity.io'da saklanır  →  urlFor(image).width(600).url() çağrılır  →  Optimize URL üretilir  →  next/image boyutlar & WebP'e çevirir  →  Kullanıcıya optimize görsel gider
